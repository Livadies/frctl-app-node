# FRCTL Hub: инструкция для проверки инвестором

## Коротко о продукте

FRCTL Hub — открытая оболочка для безопасного доступа к удалённым компьютерам и серверам, запуска изолированных рабочих сред и подключения ИИ‑workflow. Концептуально это «Steam для рабочих инструментов»: пользователь выбирает проверенный OSS‑коннектор, получает изолированную среду и контролируемый план действий.

Публичная версия показывает живой каталог GitHub + Hugging Face, интерфейс адаптеров, policy engine, локальную адаптацию и границы безопасности. Реальные разрешённые запуски и фиксированные SSH-workflow выполняет отдельный FRCTL Node 0.4 Product Alpha после системного подтверждения пользователя.

## Ссылки

- Публичная демоверсия: https://livadies-frctl-hub.static.hf.space/
- Исходники публичной демоверсии: https://huggingface.co/spaces/livadies/frctl-hub/tree/main

Регистрация и установка не требуются. Поддерживаются актуальные Chromium и Firefox.

Для обычной проверки Windows‑версии нужен архив `FRCTL-Windows-0.4.0.zip`; Python конечному пользователю не требуется. Отдельный переносимый Node доступен как `FRCTL-Node-0.4.0-win.zip`. Сам Node не размещается на Hugging Face: это намеренная граница доверия.

## Проверка основного сценария — 2 минуты

1. Откройте демоверсию.
2. Убедитесь, что указаны `LIVE CATALOG`, 48 карточек и `EXECUTION OFF`.
3. Выберите «ИИ и модели», затем введите в поиск `text-generation`.
4. Выберите карточку `RustDesk`.
5. В поле «Цель» введите `10.0.0.12:21118`.
6. В поле «Действие» выберите «Подключить» и нажмите «Сформировать dry-run».

Ожидаемый результат:

- появляется заголовок `DRY-RUN · RustDesk · 10.0.0.12:21118`;
- отображаются пять защитных шагов;
- интерфейс сообщает, что выполнение возможно только через отдельно установленный и авторизованный локальный Node;
- браузер не пытается подключиться к указанному IP.

## Негативный тест безопасности — 1 минута

Введите в поле цели:

```text
root:password@server.example
```

Ожидаемый результат: приложение отклоняет адрес, потому что credentials нельзя передавать внутри target.

Затем попробуйте:

```text
server.example?token=secret
```

Ожидаемый результат: query‑параметр также отклоняется.

## Проверка local-first — 1 минута

1. После загрузки страницы отключите интернет.
2. Повторно сформируйте план.
3. Включите «Хранить обезличенные события локально» и создайте ещё один план.

Ожидаемый результат: всё продолжает работать, потому что планировщик, история и микроагент выполняются в браузере. Данные не отправляются на сервер.

## Что демонстрирует MVP

- понятную единую оболочку для разных протоколов и сред;
- deny-by-default вместо автоматического выполнения команд;
- возможность масштабировать каталог как marketplace адаптеров;
- локальную персонализацию без передачи истории пользователя;
- основу для аудита, sandbox/VM/Kubernetes и дальнейшей сертификации.

## Проверка локального Node — 5 минут

1. Распакуйте `FRCTL-Windows-0.4.0.zip` в отдельный каталог.
2. Запустите PowerShell, выполните `.\install-windows.ps1`, затем откройте **FRCTL** через меню «Пуск».
3. В панели http://127.0.0.1:7878 выберите SSH, цель `localhost`, пользователя `demo` и нажмите «Проверить план».
4. Убедитесь, что план содержит `StrictHostKeyChecking=ask`, но не содержит пароля.
5. Не нажимая запуск, попробуйте цель `example.com`: она должна быть отклонена allowlist.
6. Для проверки на своей инфраструктуре добавьте свой сервер командой `configure-ssh-target.ps1` из README. Затем в блоке SSH Workflow выберите закреплённый сервер, PPK и «Безопасную диагностику»: после подтверждения должен появиться реальный вывод ОС. Приватный ключ автора в архив намеренно не входит.
7. Разработчик может запустить `python -m unittest discover -s tests -v` в `hub/node`: ожидается 28 успешных тестов.

Node определяет установленные OpenSSH/PuTTY, RustDesk и Docker. Проверенный Plink 0.84 для фиксированных SSH-workflow устанавливается автоматически; остальные сторонние клиенты Node не устанавливает. Реальный запуск требует нативного подтверждения; credentials вводятся только в штатном клиенте, а не в FRCTL.

## Что ещё не реализовано

- автоматическая установка всех сторонних PuTTY/RustDesk/Docker из Node;
- OAuth2‑подключение корпоративных инфраструктур;
- подписанный реестр адаптеров и supply-chain verification;
- централизованный экспорт и подпись локального журнала аудита;
- P2P‑пул вычислительных ресурсов;
- сертификация ФСТЭК.

Эти ограничения являются прозрачной границей текущей версии. Следующий технический этап — OS credential vault, OAuth2 device flow, подписанные policy bundles и управляемый реестр адаптеров.

## Критерий успешной проверки

Публичная версия считается работоспособной, если загружает живой каталог, категории и поиск работают, четыре коннектора отображаются, разрешённый target формирует dry-run, credentials отклоняются, а планировщик продолжает работать после отключения интернета. Node считается работоспособным, если локальная панель доступна только на `127.0.0.1`, формирует разрешённый план, отклоняет внешний target, выполняет закреплённый workflow только после подтверждения и все 28 тестов проходят.
